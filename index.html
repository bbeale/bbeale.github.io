Cybersecurity

What can I say? I like to break things -- the more catastropohically and cataclysmically, the better. While that aspect may be represented via my QA work, testing webapps for security vulnerabilities kicks it up several notches when compared to looking for functional defects.

The reasoning behind this lies in the methodologies a tester must use to test for each. While functional and acceptance testing require testers to assume the role of an end user and interact with the SUT accordingly, security testing involves abusing functionality to make it behave in unexpected and unsafe ways, something which a normal user would be unlikely to be or even prohibited from doing. Security tests typically do not limit the scope in the same way as functional tests, with some projects even allowing the use of special tools such as automated scanners to speed up the process.

Among the most critical of my vulnerability findings include stored XSS, SQL injections (one of which in a public facing kiosk page where any unauthenticated user who happened to pass the kiosk could have truncated an entire table critical to the core functionality of the application), and a particularly interesting session vulnerability in a learning management system with a client- and a server-side component which allowed me to modify another user's content.

As part of an academic pursuit, I recently constructed my first machine learning project -- a vulnerability scanner driven by a Generative Adversarial Network.
